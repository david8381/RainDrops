# Architecture

## Overview
Rain Math is a static browser game. Production has no bundler and no runtime dependencies: `index.html` loads `styles.css` and a single `<script type="module">` entry (`script.js`), which `import`s the `src/` modules — native ES modules, source == runtime. Because ES modules require HTTP, the app is served (`npm start` / GitHub Pages), not opened from `file://`.

Dev-only tooling: Node's built-in test runner (unit), Playwright (browser e2e), `tsc --checkJs` over the typed core (`npm run typecheck`, enforced in CI), and c8/monocart coverage.

## Design intent & module boundaries
The codebase is layered by *testability and coupling*, not by feature folders. When adding code, put it in the right layer:

- **`src/game-core.js`** — pure, DOM-free rules and game-logic. Problem generation, mastery weighting, factorization, SI; the stats/report/share **display formatters**; and pure **engine decisions** (`resolvePlacementOutcome`, the Blitz ramp curves `blitzDropSeconds`/`blitzSpeedPercent`/`blitzBombIntervalMs`, `spawnIntervalMs`, etc.). Fully unit-tested and type-checked. **Pure logic goes here.**
- **`src/player-progress.js`** — local profile + readiness/persistence. Unit-tested and type-checked. **Profile/stats logic goes here.**
- **`src/engine/`** — runtime pieces with a *real* boundary: `state.js` (the shared mutable `state` object), `predicates.js` (state queries), `splashes.js`, `laser-ship.js` (each owns its state, deps injected). **Self-contained engine subsystems go here.**
- **`src/popups/`** — modal "views": given data + callbacks (a `ctx`), build DOM. **New popups go here.**
- **`script.js`** — the **irreducibly-coupled real-time engine**: boss mode, drop management, input handling, the game loop, and UI-update orchestration. These share `state` and interact every frame, so they are kept together *on purpose* — splitting them would spread essential coupling across files behind a thick circular-import seam without making anything more maintainable. UI updates use a manual reactive pattern (refresh the affected `update*Display()` after the change; all out-of-loop, which is fine). The coupling here is essential, not accidental — don't "fix" it by moving files.

The high-leverage maintainability work was: tests + types on the core, explicit shared `state`, and extracting the parts with genuine boundaries. The remaining engine is large but cohesive and backed by the test/type net.

## Runtime Files
- `index.html`: Game markup, operation chits, controls, canvas, input bar, touch keypad, menu/test-me/login/log/donate/feedback links, feedback form, and overlays.
- `styles.css`: Dark theme, desktop layout, responsive behavior, stats popup styling, and touch-device layout.
- `script.js`: Browser state, animation loop, canvas drawing, boss-mode state machine, Test Me placement flow, Recap (share) popup, audio, DOM updates, login/stats/log/report popups, event listeners, touch keypad wiring, and gated `?test=1` hooks. The Session Log/Report popups render from swappable report data: live/local profiles are summarized into report view models, v1 shared links still decode as read-only profile snapshots, and compact v2 `#report=` links decode directly into read-only report view models. The same popups render all paths (no duplicate rendering).
- `src/game-core.js`: DOM-free game rules; an ES module (`export`) imported by `script.js` and `player-progress.js`: operation defaults, difficulty ranges, problem generation, mastery weighting, numeric normalization, SI helpers, factorization helpers, pure gated-control reason helpers, and the pure stats/report/share display formatters.
- `src/player-progress.js`: Local profile/readiness logic; an ES module that `import`s game-core and exports: profile schema, multi-profile localStorage persistence, single-profile migration, per-problem outcomes, saved Speed/Drops/Text Size settings, derived pressure-tier practice stats, saved current levels, mastery-advance records, boss-completion records, Blitz/Wave/Worksheet challenge attempts and bests, per-session/per-operation report data, level-universe sizing, readiness scoring, practice suggestions, boss recommendation flags, and the stats-tooltip builder.
- `src/text/english.js`: English text catalog; an ES module exporting `RainMathText` (welcome-menu and Tutorial copy plus button labels).
- `src/types.js`: Central JSDoc `@typedef` documentation of the core data shapes (`Drop`, `BossMode`, `OpConfig`, `Problem`, `StoredProfile`, `StoredSkill`, `SkillSummary`, etc.). No runtime code — referenced from other files via `import('./types.js').X` in JSDoc so editors/AI can resolve the data model without reverse-engineering it. **Read this first to understand what the objects flowing through the app look like.**
- `src/engine/state.js`: The engine's shared mutable runtime state as a single exported `state` object (`drops`, `score`, `bossMode`, `currentInput`, `gameSpeed`, canvas size, etc.). Engine modules import it and read/write `state.X` — this is what lets the runtime be split across files (ES module bindings are read-only for importers, so shared primitives must live on an object). `script.js` initializes the dynamic fields (e.g. `state.progressProfile = readProfile()`).
- `src/engine/splashes.js`: Drop-clear splash particles; owns the `splashes` array, takes `ctx` via `initSplashes()`.
- `src/engine/laser-ship.js`: The player cannon ship, its targeting laser, and the base shield visual; owns `laser`/`playerShip`, dependencies injected via `initShip()`.
- `src/audio.js`: Web Audio sound effects (`initAudio`, `playPop`, `playMiss`, `playWrongInput`); `audioCtx` is module-private.
- `src/popups/login-popup.js`: Login / player-profile popup as a self-contained "view" — imports profile persistence from player-progress, takes engine-stateful deps via a `ctx` object, and owns the player-manager UI for switching, creating, deleting, clearing, backing up, and restoring local profiles. `script.js` keeps a thin `openLoginPopup()` that builds the ctx. (First of the popup "views"; the remaining popups still live in `script.js`.)

The app is served over HTTP (`npm start` / GitHub Pages); native ES modules do not load from `file://`. In test mode (`?test=1`) `script.js` re-exposes `window.RainMathCore` / `window.RainMathProgress` for browser-side test instrumentation.

## Test Files

- `tests/unit/player-progress.test.js`: Deterministic tests for local profile persistence and readiness scoring.
- `tests/e2e/rain-math.spec.js`: Playwright desktop/mobile browser coverage.
- `tests/support/static-server.mjs`: Tiny local static server used by Playwright.
- `playwright.config.mjs`: Browser projects, local server, and reporter settings.
- `.github/workflows/tests.yml`: CI test workflow.

## Runtime Flow
1. The browser loads `index.html`, which loads `script.js` as a single `<script type="module">`; `script.js` imports `src/game-core.js`, `src/player-progress.js`, and `src/text/english.js` (the browser resolves the module graph).
2. `src/game-core.js` exports the pure rules; `src/player-progress.js` imports them and exports the local profile helpers; `script.js` imports both as namespaces (`RainMathCore`, `RainMathProgress`) and destructures the names it needs.
3. `script.js` creates mutable browser state, reads the active local profile from localStorage, and mirrors durable progress into legacy `problemStats` for weighted generation.
4. `init()` sizes the canvas, syncs controls/settings into the profile, starts or resumes the active local session, sets up touch UI when needed, optionally installs test hooks for `?test=1`, shows the welcome menu on first non-test visits, focuses the hidden answer input when no menu is open, and starts `requestAnimationFrame(tick)`. Session startup uses `shouldResumeSession()` from `player-progress.js`: the active profile's newest session is reused when `lastSeenAt` is within 30 minutes; otherwise a new session id is minted.
5. `tick()` spawns drops and updates motion according to the Speed and Drops controls. Speed controls fall velocity and spawn interval; Drops controls the active-drop cap. Weighted problem generation favors unmastered, low-accuracy, and under-attempted current-level problems. Space-triggered Breather mode pauses normal spawning and drop motion until all currently visible unrevealed drops are cleared.
6. Input handlers clear immediate-answer drops as soon as they match. SI and full factorization answers wait for Enter. Targeted factor drops accept one divisor at a time. Impossible submissions are tracked in a short rolling window; too many quickly triggers a transient cannon-overload cooldown that ignores input without writing ordinary mastery stats.
7. Correct, wrong, missed, and helped outcomes update both in-memory stats and the local profile during ordinary practice. Boss, Blitz, and Wave problem outcomes are excluded from ordinary mastery stats but still count in the active local session log's assessment bucket; completed challenge attempts are recorded separately as challenge records.
8. The first-visit welcome menu is built dynamically in `script.js`, with editable English copy read from `src/text/english.js`; it stores only `rainMath.welcomeSeen.v1` to remember that Play/Tutorial/Test Me placement was chosen. It shows the current player but delegates all player management to the Login popup, and the in-app Tutorial is a stepper overlay that highlights existing controls. Test Me starts from a small picker, then runs in the ordinary falling-drop playfield one problem at a time, repeating missed facts twice and using a per-level shield to advance (correct fills it and climbs a level, misses drain it, an empty shield recommends the current level); harder problem types fall more slowly. Accepting a suggested level records actual Test Me answers as ordinary practice, records lower levels as placement-advanced so the player can return to the suggested level after lowering the selector, and marks eligible lower-level facts as placed out without inventing attempts.
9. The Login popup creates, switches, deletes, clears, backs up, and restores local profiles. Switching, deleting the active profile, or restoring saves the outgoing profile, loads the selected profile's settings/progress, always starts a fresh local session log entry for that player, and clears only transient run state such as active drops and the Cleared counter. Profile deletion is handled by `deleteStoredProfile()`: non-active deletes leave the active profile alone, active deletes choose the most recently updated remaining profile, and deleting the last profile recreates a fresh default so the store is never empty. Profile backups export the active profile only as an `RMBAK1:` compressed share payload with a tamper-evidence checksum and `PROFILE_VERSION`; restore rejects damaged blobs and newer schemas, then imports older/current profiles through `importStoredProfile()` so normal migrations still apply.
10. Boss mode unlocks when Mastered reaches 100%. Mastered is the percentage of current-level problems with at least 3 attempts and at least 90% current accuracy, where current accuracy heavily weights recent weighted performance with a smaller lifetime correction. Once an operation reaches 80% but is not yet boss-ready, practice generation enters a finish-level focus mode that heavily favors the remaining unmastered current-level facts. Once mastered, a modal choice pauses the board until answered: Keep Practicing, Boss, or Next Level. Choosing Next Level records a mastery advance without recording a boss clear. Once started, boss mode pauses normal spawning, locks the current Speed/Drops snapshot plus operation/level/challenge controls, runs Wave 1 shield endurance as an open-ended drop-time ramp with fixed load, then Wave 2 as a clear-gated load ladder (a round of N problems at fixed speed that only steps up after the whole round clears), then the final Worksheet ship. The Worksheet is built from the entire current-level universe split across the four parts; nodes reveal in capped, answer-unique batches and empty trailing parts auto-collapse. Active challenge bombs, ship problem nodes, and node-backed worksheet missiles use the same input path as drops. Worksheet missiles drop only from visible remaining ship nodes; solving a missile also clears that source node. Destroying the final core records the boss attempt, advances that operation one level, and opens a victory summary with Grid and Recap actions.
11. Replay challenges unlock for any level already behind the current working level, whether it was passed by boss clear or mastery advance. They also appear for the current level once it is boss-ready, so choosing Keep Practicing after mastery does not require toggling to the next level and back. Blitz starts a shield-endurance survival run for the selected eligible level; Wave starts a simultaneous-load run for that level; Worksheet starts directly at the mothership/fact sheet and records best time without advancing content level. Higher-level challenge attempts can satisfy lower-level best lookups when they are better.

## Core Data Model
- `opConfig`: per-operation `{ enabled, difficulty, symbol, label }`, initialized from `createDefaultOpConfig()` at level 1, then overlaid with saved profile levels at startup. Operation chits read this state to update their active class, hover tooltip level text, ARIA labels, and CSS course-progress fill.
- `drops[]`: active drops with common fields `{ id, x, y, baseSpeed, text, answer, answerText, opKey, statsKey }`; factor drops add `factorOriginal`, `factorRemaining`, `factorCollected`, and `factorComplete`.
- `problemStats`: per-operation in-memory accuracy maps `{ asked, correct }`, initialized from `createProblemStats()`. The stats popup, stats hover tooltips, and falling-drop shading read from the same accuracy/evidence palette. Untested problems are black, hue runs from red through yellow to green by current accuracy, opacity/brightness increases with attempts, and Test Me placed-out problems display as green placement credit until later real attempts provide enough contrary evidence.
- `progressProfile`: durable active local profile with per-operation current level, readiness, per-problem outcome history, Test Me placement credits, saved Speed/Drops/Text Size settings, derived pressure-tier outcome buckets, mastery-advance records, boss-completion records, legacy Blitz attempts, generalized challenge attempts for Blitz/Wave/Worksheet, and a bounded `sessionLog` of recent local sittings. The active profile is stored inside the multi-profile store `rainMath.profiles.v1`; the legacy active-profile key `rainMath.profile.v1` is still written for compatibility. Untouched legacy `Local Player` data migrates into a profile named `david`; `importStoredProfile()` normalizes backed-up profiles through the same shape/migration path, replaces same-id/name collisions, and activates the restored profile; `deleteStoredProfile()` removes local profiles while preserving or recreating an active profile. Boss readiness treats unseen or under-practiced current-level problems as unmastered unless they have active placed-out credit: ordinary mastery needs at least 3 attempts and at least 90% current accuracy, while placed-out facts count as mastered until enough real attempts override the estimate. Accepting Test Me placement also writes `levelAdvances` through the level below the recommended start with result `placed-out`, preventing a temporary lower selector from blocking return to the recommended level. Practice suggestions mix weak seen problems with unseen level problems where the operation has an enumerable universe; `getFinishLevelPracticeProblems()` identifies remaining current-level blockers once readiness reaches 80%, and `script.js` gives those blockers most practice drops until boss-ready. Pressure tiers are still derived from speed for compatibility/reporting metadata, but visible practice control is Speed plus Drops rather than a preset tier selector. Challenge bests are selected by longest survival time for Blitz, highest fully cleared simultaneous load for Wave, and shortest duration for Worksheet; higher-level attempts can override lower-level displayed bests when they are better. Attempt history also keeps solved counts and pressure details for future progress graphs. Session logs separate ordinary practice stats from boss/challenge assessment stats so stressful failures do not contaminate mastery calculations. Each session also stores settings, per-operation practice/assessment stats, challenge counters, start/end mastery snapshots, level-by-level mastery snapshots, per-operation engaged duration from response times capped per item, and optional `activeMs` headline duration accrued from idle-capped activity gaps. `lastSeenAt`/`endedAt` are heartbeat-stamped during play and on `beforeunload`; startup resumes the newest active-profile session only when that timestamp is within the 30-minute sitting grace window, while displayed duration uses `activeMs` or legacy engaged-time fallback instead of wall-clock.
- `bossMode`: transient active boss, Blitz, Wave, or Worksheet replay state with the selected operation, target level, mode, locked Speed/Drops snapshot, phase, challenge type, shield/load counters, Blitz survival/drop-time metrics, the Wave 2 round-spawn and max-load counters, boss parts (each holding a universe slice of revealable worksheet nodes), falling debris, bomb timer, stun/timer state, transition sweep state, worksheet timer, and victory timer. It is not persisted; ordinary mastery is not updated during boss/challenge play, while completed challenge attempts update the local profile.
- `gameSpeed`: current Speed control value from 0-100.
- `dropLimit`: current Drops control value from 0-10. Some profile/test event fields still use the historical names `rate` or `spawnRate`; those names now mean drop-load/concurrency rather than a timed spawn-rate slider.
- `textSize`: current problem text size (`normal`, `large`, or `huge`), persisted in the profile and applied to falling drops and boss worksheet nodes.
- `isBreatherMode`: transient practice-only Spacebar state. While true, normal spawning and drop motion pause; answer processing remains live and the mode exits automatically when no visible unrevealed drops remain.
- `cannonOverloadMs` / `wrongSubmissionTimes`: transient input-control state. Impossible submissions within a short window can briefly overload the cannon; correct answers clear the recent window, and restart/profile changes clear the cooldown.
- `rainMath.welcomeSeen.v1`: small localStorage flag used only to suppress the first-visit menu after the player chooses Play, exits the Tutorial, or accepts Test Me placement. `?test=1` suppresses the menu unless `welcome=1` is also present.
- `score`: session-only count of correct drops cleared, displayed as Cleared.
- `splashes`, `laser`, `groundFlash`: visual effects state.

## Operation Types
- Basic arithmetic: `add`, `sub`, `mul`, `div`.
- Decimal shifting: `f10`. Difficulty is structural — a problem "type" is `(significant digits, power of 10, ×/÷)` with `difficulty = digits + power − 1`; the concrete number is random, so mastery accrues per type. A level holds every type with `digits + power − 1 ≤ level` (cumulative).
- Rounding & estimation: `round`, its own compatibility lane. Each level has a structural bucket universe rather than literal numbers: one cell is a rounding case (`down`, `up`, `half`, `zero`, or `carry`) scoped to that level's target place and size relationship. L1–L5 cover integer rounding to nearest ten/hundred/thousand, including crossing cases like `7 ≈ 10` and capstone carry cases; L6–L10 cover decimal rounding to nearest tenth/hundredth/thousandth, including crossing-to-zero and extra-precision review. The concrete positive number is sampled when the problem spawns, and the sampler verifies the number belongs to the cell's case. Rounding uses decimal-safe half-up logic and displays prompts as `<number> ≈ <place>`.
- SI metric conversions: `si`.
- Geometry: `shapes`, one level-gated operation (cumulative) — L1 square, L2 rectangle, L3 triangle, L4 circle, L5 cube, L6 rectangular prism, L7 cylinder, L8 sphere. 2D shapes use perimeter/area, 3D shapes use surface area/volume, and round shapes (circle, cylinder, sphere) answer as the π coefficient. Dimension combinations that would give a non-clean answer (e.g. most sphere volumes) are filtered out so every answer is an integer or a half. Levels 9–10 reuse the full L8 set.
- Powers & roots: `pow`, one level-gated operation (cumulative), 10 levels: L1 squares 2²–7², L2 squares 8²–12², L3 square roots, L4 powers of 10 (10¹–10⁶), L5 roots of 10, L6 powers of 2 (2¹–2¹⁰), L7 cubes, L8 cube roots, L9 powers of 3 (3¹–3⁶), L10 negative powers of 10 (10⁻¹–10⁻⁶). Roots use perfect powers so answers stay whole; negatives are clean terminating decimals. Superscripts/radicals via `toSuperscript` + `√/∛`. Universe/generator live in `getPowUniverse`, `makePowProblem`, `makePowProblemFromKey`.
- Prime factorization: `factor`, with full-answer and Tab-targeted stepwise modes (auto-targeted when factoring is the only operation in play, and during a factor boss for ship nodes/bombs). Difficulty is structural: `difficulty(n) = primeIndex(largest prime) + max exponent + (# primes with exponent > 1) + Ω(n) − 4`, and a level holds every composite with difficulty ≤ level + 1 (cumulative; L1 = {4, 6, 10, 15}).

## Extensibility Notes
- Add new operation defaults in `src/game-core.js`, a generator/range branch there, display labels in `script.js`, and an operation chit in `index.html`.
- Keep pure rules in `src/game-core.js`; keep DOM, drawing, audio, and browser event behavior in `script.js`.
- Keep manually edited onboarding copy in `src/text/english.js`. Future locale files should use the same object shape and be loaded as plain scripts before `script.js`, not fetched JSON, unless the direct-file workflow is intentionally dropped.
- Update `docs/Ai/CHANGELOG.md` for player-facing or architectural changes.
- Use `?test=1` hooks only in tests; do not make gameplay depend on them.
- Keep `src/player-progress.js` backend-shaped so future backend sync can replace localStorage without changing gameplay event semantics.
- Boss-readiness recommendations should remain representative-assessment estimates: count current-level problems mastered by the 3-attempt/90%-current-accuracy rule, using recent weighting as a correction for old history rather than a short streak bonus.
- Boss mode should remain an opt-in mastery test. Normal practice should stay focused on retrieval fluency rather than combat survival.
- Practice Speed and Drops should remain player-controlled and separate from content-level advancement. Do not require higher practice pressure to advance content level unless the product direction changes deliberately.
- Derived pressure-tier clears remain compatibility metadata for existing profile data. The player-facing post-mastery challenges are Blitz survival time, Wave max cleared load, and Worksheet time for a specific unlocked level.
